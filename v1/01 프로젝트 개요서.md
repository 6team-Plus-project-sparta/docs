## 2. 프로젝트 배경 및 목적

### 배경

온라인 티켓팅 시장은 순간적으로 수천~수만 명이 동시에 접속하는 **극단적 트래픽 집중** 환경이다.
인터파크, 멜론티켓, YES24 티켓 등 기존 서비스에서도 오픈 직후 서버 다운, 중복 예매, 재고 불일치 등의 문제가 반복적으로 발생한다.

### 목적

- 대규모 동시 요청에서도 **데이터 정합성을 보장**하는 티켓팅 백엔드 구현
- 캐싱 전략으로 **검색 성능을 최적화**하는 경험 습득
- 실제 운영 환경과 유사한 **CI/CD 파이프라인 및 클라우드 배포** 경험
- 백엔드 개발자로서 반드시 마주치는 **동시성·성능·운영** 문제를 직접 해결

---

## 3. 서비스 개요

**TicketFlow**는 공연(콘서트, 뮤지컬, 연극)과 스포츠(야구, 축구, 농구) 티켓을 온라인으로 예매할 수 있는 플랫폼이다.

### 핵심 가치 제안

| 가치 | 설명 |
|------|------|
| 공정한 선착순 | Redis 분산락으로 중복 예매 없는 정확한 선착순 보장 |
| 빠른 검색 | Caffeine 로컬 캐시 + Redis 캐시로 인기 검색어 즉시 응답 |
| 투명한 프로모션 | 선착순 할인 쿠폰 발급 시 정확한 수량 제한 |
| 빠른 CS 응답 | WebSocket 기반 실시간 CS 문의 채팅 |

## 4. 주요 기능 범위

### 필수 구현 (Must Have)

| 기능 | 설명 |
|------|------|
| 회원 관리 | 회원가입, 로그인 (JWT), 내 정보 조회·수정 |
| 공연·스포츠 이벤트 관리 | 이벤트 등록, 조회, 구역/좌석 관리 |
| 티켓 예매 | 좌석 선택 → 결제(Mock PG) → 예매 확정 |
| 선착순 쿠폰 발급 | 한정 수량 할인 쿠폰 선착순 발급 |
| 검색 기능 | 이벤트명·장르 LIKE 검색 + 페이징 |
| 인기 검색어 | Redis ZSet 기반 실시간 인기 검색어 Top 10 |
| 로컬 캐싱 | Caffeine 캐시 v1/v2 API 비교 |
| 동시성 제어 | Redis 분산락으로 티켓·쿠폰 재고 정합성 보장 |

### 도전 구현 (Should Have)

| 기능 | 설명 |
|------|------|
| Redis 캐싱 | v2 API를 Redis Cache-Aside 패턴으로 업그레이드 |
| DB 인덱스 최적화 | EXPLAIN 기반 쿼리 분석 및 복합 인덱스 적용 |
| CS 채팅 | WebSocket(STOMP) 기반 고객↔관리자 실시간 문의 채팅 |
| CI/CD 자동화 | GitHub Actions + ECR + EC2 SSM 자동 배포 |

### 제외 범위 (Out of Scope)

- 실제 PG 연동 (토스페이먼츠 Mock으로 대체)
- 실시간 좌석 배치도 렌더링
- 정산·환불 플로우
- 모바일 앱

## 5. 팀 구성 및 역할 분담

| 팀원 | 담당 도메인 | 주요 책임 |
|------|------------|-----------|
| 팀원 A | **회원 (User)** | 회원가입/로그인, JWT 인증, 내 예매 내역 |
| 팀원 B | **이벤트 (Event)** | 공연·스포츠 이벤트 CRUD, 구역/좌석 관리, 검색·캐싱 |
| 팀원 C | **예매 (Booking)** | 티켓 예매 플로우, 동시성 제어, Mock 결제 연동 |
| 팀원 D | **프로모션 (Promotion)** | 선착순 쿠폰 발급, 동시성 제어, CS 채팅 |
| 팀원 E | **인프라 (Infra)** | Docker, AWS 구성, GitHub Actions CI/CD, DB 인덱스 최적화 |

> 팀원 E는 인프라 구성이 완료된 후 팀원 B의 인덱스 최적화 작업을 함께 지원한다.

## 6. 기술 스택

### Backend

| 분류 | 기술 | 선택 이유 |
|------|------|-----------|
| 언어/프레임워크 | Java 17 + Spring Boot 3.x | 팀 공통 학습 스택, Virtual Thread 지원 |
| ORM | Spring Data JPA + QueryDSL | 동적 검색 쿼리, 타입 안전 |
| 인증 | Spring Security + JWT | Stateless 인증, 확장성 |
| 로컬 캐시 | Caffeine | Spring Cache 추상화 연동, 고성능 |
| 분산 캐시/락 | Redis (Lettuce, Redisson) | 분산 환경 데이터 공유, Pub/Sub |
| 실시간 통신 | WebSocket + STOMP | CS 채팅 양방향 통신 |

### Database

| 분류 | 기술 | 선택 이유 |
|------|------|-----------|
| RDBMS | MySQL 8 | 트랜잭션 보장, InnoDB 락 지원 |
| 캐시 스토어 | Redis 7 | 인기 검색어(ZSet), 분산 락, 세션 |

### Infra

| 분류 | 기술 |
|------|------|
| 컨테이너 | Docker + Docker Compose |
| 클라우드 | AWS EC2 · RDS · ElastiCache · ECR |
| CI/CD | GitHub Actions |
| 시크릿 관리 | GitHub Secrets + AWS Parameter Store |

## 7. 프로젝트 일정

```

  Day 1-3  : SA 문서 작성, ERD 확정, API 명세 작성, 튜터 피드백

Week 3 (QA + 도전 + 배포)
  Day 15-17: Redis 캐시 전환, DB 인덱스 최적화, 성능 테스트
  Day 18-19: AWS 배포, CI/CD 파이프라인 구성
  Day 20-21: QA, 버그 수정, README 작성, 발표 준비
```
