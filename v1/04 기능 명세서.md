# ⚙️ 기능 명세서

## 1. 회원 관리 (User Domain)

### 1-1. 회원가입

| 항목 | 내용 |
|------|------|
| 입력 | email, password, nickname, phone |
| 처리 | 이메일 중복 검사 → BCrypt 비밀번호 해시 → User 저장 |
| 출력 | 201 Created, userId |
| 예외 | 이메일 중복 409 / 형식 오류 400 |
| 비고 | 기본 ROLE: USER |

### 1-2. 로그인

| 항목 | 내용 |
|------|------|
| 입력 | email, password |
| 처리 | 이메일 조회 → 비밀번호 검증 → Access Token(15분) + Refresh Token(7일) 발급 |
| 출력 | 200 OK, accessToken, refreshToken |
| 예외 | 이메일 없음 / 비밀번호 불일치 → 401 Unauthorized |
| 비고 | Refresh Token은 Redis에 저장 (Key: userId) |

### 1-3. 토큰 재발급

| 항목 | 내용 |
|------|------|
| 입력 | refreshToken (헤더) |
| 처리 | Redis에서 Refresh Token 검증 → 새 Access Token 발급 |
| 출력 | 200 OK, accessToken |

## 2. 이벤트 관리 (Event Domain)

### 2-1. 이벤트 등록 (Admin)

| 항목 | 내용 |
|------|------|
| 입력 | title, description, genre, venue, startAt, endAt, posterImageUrl |
| 처리 | Admin 권한 확인 → Event 저장 → 구역(Section) 연결 |
| 출력 | 201 Created, eventId |

### 2-2. 이벤트 상세 조회

| 항목 | 내용 |
|------|------|
| 입력 | eventId (PathVariable) |
| 처리 | Event + Section + Seat Fetch Join 조회 |
| 출력 | 200 OK, 이벤트 정보 + 구역별 잔여 좌석 수 |
| 비고 | N+1 방지: Section List Fetch Join |

### 2-3. 이벤트 검색 v1 (캐시 미적용)

| 항목 | 내용 |
|------|------|
| 입력 | keyword, genre, startDate, endDate, page(0), size(10) |
| 처리 | QueryDSL 동적 쿼리 (BooleanExpression) → LIKE 검색 → Page 반환 |
| 출력 | 200 OK, Page\<EventSummaryResponse\> |
| SQL | `WHERE title LIKE '%keyword%' AND genre = ?` |
| 비고 | 검색 시 Redis ZSet ZINCRBY keyword 1 |

### 2-4. 이벤트 검색 v2 (Caffeine 캐시 적용)

| 항목 | 내용 |
|------|------|
| 입력 | keyword, genre, startDate, endDate, page, size |
| 처리 | `@Cacheable(value="eventSearch", key="#keyword + #genre + #page")` |
| 캐시 설정 | TTL 5분, maximumSize 1000 |
| 출력 | 200 OK, Page\<EventSummaryResponse\> (캐시 HIT 시 DB 미조회) |
| 비고 | 이벤트 수정 시 `@CacheEvict(value="eventSearch", allEntries=true)` |

### 2-5. 인기 검색어 조회

| 항목 | 내용 |
|------|------|
| 처리 | Redis `ZREVRANGE popular:search 0 9 WITHSCORES` → Top 10 반환 |
| 캐시 | Caffeine TTL 5분 (`@Cacheable`) |
| 출력 | 200 OK, List\<PopularKeywordResponse\> (keyword, score) |
| 집계 기간 | 실시간 (별도 만료 없음, 추후 일별 초기화 Scheduler 추가 가능) |

## 3. 예매 (Booking Domain)

### 3-1. 좌석 선택 및 임시 선점

| 항목 | 내용 |
|------|------|
| 입력 | eventId, seatId |
| 처리 | 좌석 상태 확인(AVAILABLE) → Redis `SETNX seat:{seatId} userId TTL 300` |
| 출력 | 200 OK, holdToken (선점 확인용) |
| 예외 | 이미 선점/예매된 좌석 409 |
| 비고 | 5분 내 예매 미완료 시 자동 해제 |

### 3-2. 예매 요청 (분산락 적용) ⭐

| 항목 | 내용 |
|------|------|
| 입력 | seatId, couponId(optional) |
| 처리 흐름 | Redis Lock 획득 → 좌석 상태 재확인 → 쿠폰 유효성 검사 → Booking 생성(PENDING) → 재고 1 차감 → Lock 해제 |
| Lock Key | `lock:seat:{seatId}` |
| Lock TTL | 3초 |
| 재시도 전략 | 즉시 실패 (Fail Fast) — 티켓팅 특성상 대기보다 빠른 실패 응답이 UX에 유리 |
| 출력 | 201 Created, bookingId, totalAmount |
| 예외 | 락 획득 실패 409 / 재고 없음 400 |

### 3-3. 결제 (Mock PG)

| 항목 | 내용 |
|------|------|
| 입력 | bookingId, paymentMethod, cardNumber(masked) |
| 처리 | Mock PG API 호출 (토스페이먼츠 구조 모방) → 랜덤 성공/실패 (성공률 95%) → Booking status PENDING → CONFIRMED |
| 출력 | 200 OK, paymentId, status |
| 실패 처리 | 결제 실패 시 Booking CANCELLED, 좌석 AVAILABLE 복구 |

### 3-4. 예매 취소

| 항목 | 내용 |
|------|------|
| 입력 | bookingId |
| 처리 | 본인 확인 → 공연 7일 전까지만 취소 가능 → Booking CANCELLED → 좌석 AVAILABLE → 환불(Mock) |
| 출력 | 200 OK |

## 4. 프로모션 (Promotion Domain)

### 4-1. 쿠폰 등록 (Admin)

| 항목 | 내용 |
|------|------|
| 입력 | name, discountType(PERCENT/FIXED), discountValue, totalQuantity, expiredAt |
| 처리 | Coupon 저장, 잔여 수량 = totalQuantity |
| 출력 | 201 Created, couponId |

### 4-2. 선착순 쿠폰 발급 (분산락 적용) ⭐

| 항목 | 내용 |
|------|------|
| 입력 | couponId |
| 처리 흐름 | Redis Lock 획득 → 잔여 수량 확인 → 중복 발급 확인 → CouponIssue 생성 → remainingQuantity -1 → Lock 해제 |
| Lock Key | `lock:coupon:{couponId}` |
| Lock TTL | 3초 |
| 재시도 전략 | Retry with backoff (100ms 간격, 최대 3회) — 쿠폰은 약간의 대기 허용 |
| 출력 | 201 Created, couponIssueId |
| 예외 | 재고 소진 400 / 중복 발급 400 |

> **왜 티켓과 쿠폰의 재시도 전략이 다른가?**
> 티켓은 오픈 순간 수천 명이 동시에 시도하므로 빠른 실패가 UX에 유리하다.
> 쿠폰은 수십~수백 명 수준이므로 짧은 재시도로 더 많은 사용자에게 기회를 줄 수 있다.

## 5. CS 채팅 (Chat Domain)

### 5-1. 문의 채팅방 생성

| 항목 | 내용 |
|------|------|
| 입력 | title (문의 제목) |
| 처리 | ChatRoom 생성 (status: WAITING, userId 연결) |
| 출력 | 201 Created, roomId |

### 5-2. WebSocket 연결 및 인증

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `ws://host/ws-chat` |
| 인증 | STOMP CONNECT 헤더의 Authorization: Bearer {JWT} → ChannelInterceptor에서 검증 → Principal 설정 |
| 구독 | `SUBSCRIBE /sub/chat/{roomId}` |
| 발행 | `SEND /pub/chat/{roomId}` |

### 5-3. 메시지 송수신

| 항목 | 내용 |
|------|------|
| 입력 | roomId (destination), content |
| 처리 | Principal에서 발신자 추출 → ChatMessage DB 저장 → `/sub/chat/{roomId}` 브로드캐스트 |
| 출력 | ChatMessageResponse (senderId, content, createdAt) |

### 5-4. 채팅 내역 조회 (커서 기반 페이징)

| 항목 | 내용 |
|------|------|
| 입력 | roomId, lastMessageId(optional), size(20) |
| 처리 | `WHERE id < :lastMessageId ORDER BY id DESC LIMIT :size` |
| 출력 | List\<ChatMessageResponse\> |
| 비고 | Offset 방식 대신 커서 방식 → 대량 메시지에서도 일정한 성능 |

### 5-5. 문의 상태 변경 (Admin 전용)

| 항목 | 내용 |
|------|------|
| 입력 | roomId, newStatus |
| 처리 | Admin 권한 확인 → 상태 전이 유효성 검사 → ChatRoom status 업데이트 |
| 상태 전이 규칙 | WAITING → IN_PROGRESS → COMPLETED (역방향 불가) |
| 출력 | 200 OK |

## 6. 공통 기능

### 6-1. 예외 처리

모든 API는 `@RestControllerAdvice` 글로벌 예외 핸들러를 통해 일관된 에러 응답 반환

```json
{
  "status": 400,
  "error": "BAD_REQUEST",
  "message": "쿠폰이 모두 소진되었습니다.",
  "timestamp": "2025-07-01T14:00:05"
}
```

### 6-2. 인증 필터 흐름

```
HTTP 요청
  → JwtAuthFilter (OncePerRequestFilter)
  → Bearer Token 추출 및 검증
  → SecurityContextHolder에 Authentication 설정
  → Controller 진입
```

### 6-3. 캐시 전략 요약

| 캐시 대상 | 구현체 | TTL | Evict 조건 |
|-----------|--------|-----|------------|
| 이벤트 검색 v1 | Caffeine | 5분 | 이벤트 수정/삭제 시 |
| 인기 검색어 | Caffeine | 5분 | 자동 만료 |
| 이벤트 검색 v2 (도전) | Redis | 10분 | 이벤트 수정/삭제 시 |
