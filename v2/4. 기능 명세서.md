## 문서 구조 안내

이 문서는 유스케이스 명세서(UC-001 ~ UC-012)를 **개발자가 바로 구현할 수 있는 기능 단위**로 세분화한다.
각 기능(FN)은 하나의 API 엔드포인트 또는 독립된 비즈니스 로직에 대응된다.

### 기능 ID 체계

```
FN-[도메인 약어]-[순번]

도메인 약어:
  AUTH  = 회원/인증
  EVT   = 이벤트 관리
  SRCH  = 검색
  SEAT  = 좌석/Hold
  BK    = 예매(Order & Booking)
  CPN   = 쿠폰/프로모션
  CHAT  = CS 채팅
```

## 1. 회원/인증 도메인 (AUTH)

> **담당:** 팀원 A / **연결 UC:** UC-001, UC-002, UC-003

---

### FN-AUTH-01: 회원가입 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-001 |
| 담당 팀원 | A |
| API | `POST /api/auth/signup` |

**기능 설명**

사용자가 이메일, 비밀번호, 닉네임을 입력하여 TicketFlow에 가입한다.

**입력값**

| 필드 | 타입 | 필수 | 검증 규칙 |
|------|------|:---:|-----------|
| email | String | ✅ | 이메일 형식 (`@` 포함), 최대 100자 |
| password | String | ✅ | 8자 이상, 영문 + 숫자 포함 |
| nickname | String | ✅ | 2~20자, 특수문자 불가 |

**처리 로직**

1. 입력값 유효성 검증 (Bean Validation)
2. 이메일 중복 조회 (`SELECT COUNT(*) FROM user WHERE email = ?`)
3. 비밀번호 BCrypt 암호화
4. User 엔티티 저장 (role = `USER`)
5. 성공 응답 반환

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 201 Created | `{ "userId": 1, "email": "..." }` |
| 이메일 중복 | 409 Conflict | "이미 가입된 이메일입니다." |
| 유효성 실패 | 400 Bad Request | 필드별 에러 메시지 |

**비고**

- 관리자 계정은 직접 DB INSERT 또는 별도 시드 데이터로 생성 (회원가입 API로 생성 불가)
- 비밀번호 평문 로그 출력 금지

---

### FN-AUTH-02: 로그인 (JWT 발급) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-002 |
| 담당 팀원 | A |
| API | `POST /api/auth/login` |

**기능 설명**

이메일/비밀번호 인증 후 JWT Access Token을 발급한다.

**입력값**

| 필드 | 타입 | 필수 |
|------|------|:---:|
| email | String | ✅ |
| password | String | ✅ |

**처리 로직**

1. 이메일로 사용자 조회
2. BCrypt 비밀번호 검증
3. 인증 성공 시 JWT Access Token 생성
    - payload: `{ "userId": 1, "role": "USER", "iat": ..., "exp": ... }`
    - 만료: 1시간
    - 서명: HS256 + 서버 Secret Key
4. 응답 헤더 `Authorization: Bearer {token}` 또는 Body에 토큰 반환

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 성공 | 200 OK | `{ "accessToken": "eyJ..." }` |
| 인증 실패 | 401 Unauthorized | "이메일 또는 비밀번호가 올바르지 않습니다." |

**비고**

- 인증 실패 시 이메일 존재 여부를 구분하지 않음 (보안 원칙)
- Refresh Token은 이번 스코프에서 제외 (Out of Scope)

## 2. 이벤트 관리 도메인 (EVT)

> **담당:** 팀원 B / **연결 UC:** UC-004

---

### FN-EVT-01: 이벤트 등록 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-004 |
| 담당 팀원 | B |
| API | `POST /api/admin/events` |
| 인증 | 🔒 JWT 필수 (ADMIN) |

**기능 설명**

관리자가 공연/스포츠 이벤트를 등록하고, 구역 및 좌석을 일괄 생성한다.

**입력값**

| 필드 | 타입 | 필수 | 검증 규칙 |
|------|------|:---:|-----------|
| title | String | ✅ | 1~200자 |
| category | Enum | ✅ | CONCERT, MUSICAL, THEATER, SPORTS |
| venueId | Long | ✅ | 존재하는 Venue ID |
| eventDate | LocalDateTime | ✅ | 현재 시간 이후 |
| description | String | ❌ | 최대 2,000자 |
| thumbnailUrl | String | ❌ | URL 형식 |
| sections | List | ✅ | 최소 1개 |
| └ sectionName | String | ✅ | 구역명 |
| └ price | Integer | ✅ | 0 이상 |
| └ totalSeats | Integer | ✅ | 1 이상 |

**처리 로직**

1. ADMIN 권한 검증
2. venueId 존재 여부 확인
3. Event 엔티티 저장
4. 각 Section 엔티티 저장
5. 각 Section의 `totalSeats` 수만큼 Seat 엔티티 일괄 생성 (`seat_number = {sectionName}-{순번}`)
6. 배치 INSERT 최적화 (`hibernate.jdbc.batch_size=50`)

**응답**

| 상황 | HTTP 상태 | 응답 |
|------|-----------|------|
| 성공 | 201 Created | `{ "eventId", "title", "totalSeats" }` |
| 권한 없음 | 403 Forbidden | "관리자 권한이 필요합니다." |
| 과거 일시 | 400 Bad Request | "이벤트 일시는 현재 이후여야 합니다." |

**비고**

- 이벤트 수정/삭제는 이번 스코프에서 제외

## 3. 검색 도메인 (SRCH)

> **담당:** 팀원 B / **연결 UC:** UC-005, UC-006

---

### FN-SRCH-01: 이벤트 검색 v1 (캐시 미적용) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-005 |
| 담당 팀원 | B |
| API | `GET /api/v1/events/search` |
| 인증 | 🔓 불필요 |

**기능 설명**

키워드 + 필터 조건으로 이벤트를 검색한다. 캐시를 적용하지 않고 매 요청마다 DB를 직접 조회한다.
**v2와 동일한 인터페이스**를 유지하여 성능 비교가 가능하도록 한다.

**입력값 (Query Parameters)**

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|:---:|------|
| keyword | String | ❌ | 이벤트명 LIKE 검색 |
| category | Enum | ❌ | CONCERT, MUSICAL, THEATER, SPORTS |
| startDate | LocalDate | ❌ | 검색 시작일 |
| endDate | LocalDate | ❌ | 검색 종료일 |
| minPrice | Integer | ❌ | 최소 가격 |
| maxPrice | Integer | ❌ | 최대 가격 |
| page | Integer | ❌ | 기본값 0 |
| size | Integer | ❌ | 기본값 20 |
| sort | String | ❌ | eventDate,asc / price,asc 등 |

**처리 로직**

1. 검색어가 존재하면 Redis ZSet에 `ZINCRBY search-keywords {keyword} 1` (인기 검색어 집계)
2. QueryDSL BooleanBuilder로 동적 WHERE 조건 조합
    - keyword: `event.title LIKE %keyword%`
    - category: `event.category = ?`
    - startDate/endDate: `event.eventDate BETWEEN ? AND ?`
    - minPrice/maxPrice: `section.price BETWEEN ? AND ?` (서브쿼리 또는 JOIN)
3. 페이징 + 정렬 적용
4. 결과 DTO 변환 후 반환

**응답**

| 상황 | HTTP 상태 |
|------|-----------|
| 성공 | 200 OK (Page 응답) |
| 결과 없음 | 200 OK (빈 Page, `"검색 결과가 없습니다."` 메시지 포함) |

**비고**

- 이 API는 성능 비교 기준선(Baseline)이므로, 캐시를 절대 적용하지 않음
- 응답 시간을 로그로 기록하여 v2와 비교 가능하도록 함

### FN-SRCH-02: 이벤트 검색 v2 (Caffeine 로컬 캐시) 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-005 |
| 담당 팀원 | B |
| API | `GET /api/v2/events/search` |
| 인증 | 🔓 불필요 |

**기능 설명**

FN-SRCH-01과 동일한 입력/출력 인터페이스를 가지되, **Caffeine 로컬 캐시**를 적용한다.

**캐시 설계**

| 항목 | 설정값 | 이유 |
|------|--------|------|
| 캐시명 | `event-search` | |
| 캐시 키 | `keyword:category:startDate:endDate:minPrice:maxPrice:page:size:sort` | 검색 조건 전체를 키로 사용, 동일 조건 재검색 시 캐시 히트 |
| TTL | 5분 | 이벤트 정보는 자주 변경되지 않으나, 너무 오래 캐시하면 신규 이벤트 노출 지연 |
| maximumSize | 1,000 entries | 서버 메모리 보호 (약 50MB 이하 유지 추정) |
| Eviction | LRU (Least Recently Used) | 자주 검색되는 조건이 캐시에 유지 |

**처리 로직**

1. 검색어가 존재하면 Redis ZSet ZINCRBY (인기 검색어 집계 — v1과 동일)
2. `@Cacheable(value = "event-search", key = "#cacheKey")` 어노테이션 적용
3. 캐시 히트 → DB 조회 없이 즉시 반환
4. 캐시 미스 → MySQL 조회 → 결과를 Caffeine 캐시에 저장 → 반환

**Spring 설정 (application.yml)**

```yaml
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=1000,expireAfterWrite=5m
```

**응답**

v1과 동일한 형식. 응답 헤더에 `X-Cache: HIT` 또는 `X-Cache: MISS`를 추가하여 캐시 적중 여부를 프론트엔드/테스트에서 확인 가능하도록 한다.

**비고**

- v1과 v2의 응답 속도를 비교하는 테스트 코드 또는 로그 출력 필수
- 캐시 무효화: 이벤트 등록/수정 시 `@CacheEvict(value = "event-search", allEntries = true)`

### FN-SRCH-03: 이벤트 검색 v2 Redis 캐시 전환 🔵

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-005 |
| 담당 팀원 | B |
| API | `GET /api/v2/events/search` (동일 엔드포인트) |
| 인증 | 🔓 불필요 |

**기능 설명**

FN-SRCH-02의 Caffeine 로컬 캐시를 **Redis Cache-Aside 패턴**으로 교체한다.
엔드포인트 URL과 입출력은 변경 없이, 캐시 저장소만 Redis로 전환한다.

**Cache-Aside 패턴 흐름**

```
요청 → Redis GET(캐시 키)
       ├─ HIT → 즉시 반환
       └─ MISS → MySQL 조회 → Redis SET(캐시 키, TTL 5분) → 반환
```

**캐시 설계**

| 항목 | 설정값 |
|------|--------|
| 캐시 키 | `search:{keyword}:{category}:{startDate}:{endDate}:{minPrice}:{maxPrice}:{page}:{size}:{sort}` |
| TTL | 5분 |
| 직렬화 | JSON (GenericJackson2JsonRedisSerializer) |
| 무효화 | 이벤트 등록 시 `search:*` 패턴 삭제 (`SCAN` + `DEL`, `KEYS` 사용 금지) |

**Caffeine 대비 개선점**

| 항목 | Caffeine | Redis |
|------|----------|-------|
| 캐시 범위 | 서버 1대 로컬 | 전체 서버 공유 |
| Scale-out 시 | 서버 간 캐시 불일치 | 캐시 일관성 보장 |
| 메모리 관리 | JVM 힙 사용 | 외부 Redis 메모리 |
| 네트워크 비용 | 없음 | Redis 왕복 1~2ms |

**비고**

```json
{

        { "seatId": 1, "seatNumber": "A구역-001", "status": "AVAILABLE" },
        { "seatId": 2, "seatNumber": "A구역-002", "status": "ON_HOLD" },
        { "seatId": 3, "seatNumber": "A구역-003", "status": "CONFIRMED" }
      ]
    }
  ]
}
```

**비고**

- ON_HOLD 상태는 Redis 존재 여부로만 판단 (DB에 ON_HOLD 상태를 별도 저장하지 않음)

### FN-SEAT-02: 좌석 선택 (Hold 획득) ⭐ 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-007 |
| 담당 팀원 | C |
| API | `POST /api/events/{eventId}/seats/{seatId}/hold` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

사용자가 선택한 좌석을 5분간 임시 점유(Hold)한다. 동시에 동일 좌석을 선택하는 다수 사용자 중 단 1명만 성공한다.

**처리 로직**

```
1. JWT에서 userId 추출
2. [어뷰징 체크] Redis SCARD user-holds:{userId}
   └ 4 이상이면 → 400 "최대 4석까지만 선택할 수 있습니다." (즉시 반환)
3. [분산락 획득] Redis SETNX lock:seat:{eventId}:{seatId} {uuid} EX 3
   └ 실패 → 409 "다른 사용자가 처리 중입니다." (Fail Fast)
4. [락 내부] Redis EXISTS hold:{eventId}:{seatId}
   └ 이미 존재 → 락 해제 → 409 "이미 선점된 좌석입니다."
5. [락 내부] MySQL에서 seat.status 확인
   └ CONFIRMED → 락 해제 → 409 "이미 예매된 좌석입니다."
6. [락 내부] Redis SET hold:{eventId}:{seatId} {userId} EX 300
7. [락 내부] Redis SADD user-holds:{userId} {eventId}:{seatId}
8. [락 해제] Lua Script: 
   if redis.call("get", KEYS[1]) == ARGV[1] then 
     return redis.call("del", KEYS[1]) 
   end
9. holdToken 생성 (UUID v4)
10. 응답: { "holdToken": "...", "expiresAt": "...(5분 후)" }
```

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| Hold 성공 | 200 OK | `{ "holdToken", "expiresAt" }` |
| Hold 수 초과 | 400 Bad Request | "최대 4석까지만 선택할 수 있습니다." |
| 락 획득 실패 | 409 Conflict | "다른 사용자가 처리 중입니다." |
| 이미 선점됨 | 409 Conflict | "이미 선점된 좌석입니다." |
| 이미 예매됨 | 409 Conflict | "이미 예매된 좌석입니다." |

**동시성 제어 상세**

| 항목 | 설정 |
|------|------|
| 락 키 | `lock:seat:{eventId}:{seatId}` |
| 락 TTL | 3초 (락 내부 처리가 3초를 초과하면 안 됨) |
| 락 값 | UUID (본인 락만 해제하기 위한 식별자) |
| 실패 전략 | Fail Fast (재시도 없음, 티켓팅 특성상 즉시 응답이 UX에 유리) |
| 락 해제 | Lua Script로 원자적 삭제 (GET + DEL 분리 시 Race Condition 위험) |

**중요 비고**: Hold는 **개별 좌석 단위**로 Redis에 관리한다. 여러 좌석을 동시에 Hold한 후 결제는 BK 도메인(ORDER)에서 일괄 처리한다.

### FN-CPN-02: 선착순 쿠폰 발급 ⭐ 🟢

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-011 |
| 담당 팀원 | D |
| API | `POST /api/coupons/{couponId}/issue` |
| 인증 | 🔒 JWT 필수 |

**기능 설명**

사용자가 한정 수량 쿠폰을 선착순으로 발급받는다. 동시 500명이 요청해도 정확히 설정된 수량만 발급된다.

**처리 로직**

```
1. JWT에서 userId 추출
2. [중복 발급 체크] UserCoupon 테이블에 해당 userId + couponId 존재 여부 확인
   └ 이미 존재 → 409 "이미 발급받은 쿠폰입니다."
3. [잔여 수량 차감] Redis를 Primary로 사용 (Lua Script 또는 Atomic DECR)
   - MySQL remaining_quantity는 Redis 유실 대비 백업 용도로만 유지
   - 발급 성공 시 Redis + MySQL 동기화 (선택적)
   └ 0이면 → 409 "쿠폰이 모두 소진되었습니다."
4. UserCoupon INSERT (userId, couponId, status=ISSUED, issuedAt)
5. 응답: "쿠폰이 발급되었습니다!"
```

**동시성 제어 상세**

| 항목 | 설정 |
|------|------|
| 수량 차감 | Redis Atomic DECR 또는 Lua Script (Primary) |
| MySQL 동기 | Redis 유실 대비 백업 (선택적 동기화) |
| 실패 전략 | Retry with backoff (최대 3회) |
| 실패 전략 선택 이유 | 좌석과 달리 쿠폰은 "어떤 쿠폰을 받느냐"가 아니라 "받느냐 못 받느냐"이므로, 잠깐 기다려서라도 받는 것이 UX에 유리 |

**응답**

| 상황 | HTTP 상태 | 응답 메시지 |
|------|-----------|-------------|
| 발급 성공 | 200 OK | "쿠폰이 발급되었습니다!" |
| 중복 발급 | 409 Conflict | "이미 발급받은 쿠폰입니다." |
| 수량 소진 | 409 Conflict | "쿠폰이 모두 소진되었습니다." |
| 락 획득 실패 | 503 Service Unavailable | "요청이 많아 처리할 수 없습니다." |
| 발급 시작 전 | 400 Bad Request | "쿠폰 발급 시간이 아닙니다." |

### FN-CHAT-02: WebSocket 연결 및 메시지 송수신 🔵

| 항목 | 내용 |
|------|------|
| 연결 UC | UC-012 |
| 담당 팀원 | D |
| 엔드포인트 | WebSocket `ws://host/ws-stomp` |

**기능 설명**

STOMP 프로토콜을 사용하여 사용자와 관리자 간 실시간 메시지를 송수신한다.

**STOMP 설계**

| 항목 | 경로 | 설명 |
|------|------|------|
| 연결 | `/ws-stomp` | WebSocket 핸드셰이크 엔드포인트 |
| 구독 (사용자) | `/sub/chat/room/{chatRoomId}` | 해당 채팅방 메시지 수신 |
| 구독 (관리자) | `/sub/chat/rooms` | 새 채팅방 알림 수신 |
| 전송 (사용자) | `/pub/chat/message` | 메시지 발행 |
| 전송 (관리자) | `/pub/chat/message` | 메시지 발행 (동일) |

**메시지 형식**

```json
{
  "chatRoomId": 1,
  "senderId": 5,
  "senderRole": "USER",
  "content": "결제는 됐는데 예매 내역에 티켓이 없어요.",
  "sentAt": "2025-06-13T20:10:00"
}
```

**처리 로직**

1. 클라이언트가 `/ws-stomp`에 WebSocket 연결
2. STOMP CONNECT 프레임 전송 (JWT 토큰 포함)
3. 서버가 JWT 검증 후 연결 수락
4. 사용자가 `/sub/chat/room/{chatRoomId}` 구독
5. 메시지 전송 시 `/pub/chat/message`로 발행
6. 서버의 `@MessageMapping` 핸들러가 수신:
    - ChatMessage 엔티티 DB 저장
    - `/sub/chat/room/{chatRoomId}` 구독자에게 브로드캐스트

**비고**

- Spring 내장 Simple Broker 사용 (외부 메시지 브로커 불필요)
- 연결 끊김 시 STOMP reconnect 자동 재연결 (클라이언트 측 구현)
- 채팅 이력 조회 API (`GET /api/chat/rooms/{chatRoomId}/messages`)를 별도로 제공하여 재연결 시 이전 메시지 복원

## 9. 핵심 설계 원칙 (ERD v4 연계)

- **Hold**: 개별 좌석 단위 (Redis TTL)
- **주문 단위**: 여러 좌석 + 쿠폰 적용은 **하나의 ORDER**로 처리
- **결제 단위**: ORDER (total_amount, discount_amount, final_amount 관리)
- **좌석 상태**: SEAT.status는 CONFIRMED / CANCELLED만 관리 (ON_HOLD는 Redis)
- **멀티 좌석**: 최대 4석까지 하나의 ORDER로 처리 가능
