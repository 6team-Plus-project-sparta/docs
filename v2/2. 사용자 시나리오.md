## 1. 페르소나 정의

> TicketFlow의 핵심 사용자 유형을 2가지로 정의한다.
> 각 페르소나는 실제 시나리오 설계의 기준점이 되며, API 명세·ERD 설계 시에도 이 관점을 유지한다.

---

### 페르소나 A — 일반 회원 "김지수" 🎵

| 항목 | 내용 |
|------|------|
| 이름 | 김지수 |
| 나이 | 27세 |
| 직업 | 직장인 (마케터) |
| 거주지 | 서울 마포구 |
| 디바이스 | 스마트폰(주) + 노트북(보조) |
| 티켓팅 경험 | 중급 — 인터파크·멜론티켓 이용 경험 있음, 오픈런 실패 경험 다수 |

**행동 패턴**
- 좋아하는 아이돌 콘서트 오픈 당일 점심시간을 이용해 모바일로 접속한다.
- 인기 공연은 오픈 5분 전부터 새로고침을 반복한다.
- 쿠폰이 있으면 반드시 적용하고 결제한다.
- 예매 실패 시 인기 검색어를 통해 다른 공연을 탐색한다.
- 예매 후 문제가 생기면 CS 채팅으로 빠른 답변을 원한다.

**목표**
- 원하는 좌석을 공정하게 선점하고 싶다.
- 쿠폰 혜택을 놓치지 않고 싶다.
- 예매 관련 문의를 빠르게 해결하고 싶다.

**불만 / 페인포인트**
- 결제까지 완료했는데 "이미 예매된 좌석"이라는 에러를 경험한 적 있다.
- 인기 검색어가 실시간으로 안 바뀌어서 트렌드 파악이 느리다.
- CS 문의 후 답변이 언제 오는지 알 수 없어 답답하다.

---

### 페르소나 B — 관리자 "이민준" 🛠️

| 항목 | 내용 |
|------|------|
| 이름 | 이민준 |
| 나이 | 34세 |
| 직업 | TicketFlow 운영팀 |
| 주요 업무 | 이벤트 등록·관리, 쿠폰 발급 운영, CS 문의 처리 |
| 디바이스 | 노트북(주) |
| 기술 수준 | 비개발자, 관리자 페이지 중심 업무 |

**행동 패턴**
- 공연사로부터 이벤트 정보를 받아 시스템에 직접 등록한다.
- 오픈 전날 선착순 쿠폰을 세팅하고, 발급 수량을 관리한다.
- 오픈 당일 CS 채팅 문의를 실시간으로 확인하고 처리한다.
- 대기 중인 문의를 처리 중 → 완료 순서로 상태를 업데이트한다.

**목표**
- 이벤트·쿠폰을 실수 없이 정확하게 등록하고 싶다.
- CS 문의를 빠짐없이 처리해 고객 만족도를 높이고 싶다.
- 처리 중인 문의와 대기 중인 문의를 한눈에 파악하고 싶다.

**불만 / 페인포인트**
- 쿠폰 수량이 초과 발급되는 사고가 발생한 적 있다.
- CS 문의가 몰릴 때 어떤 문의가 처리됐는지 파악하기 어렵다.

## 2. 사용자 시나리오

> 각 시나리오는 **정상 흐름(Happy Path)**과 **예외 흐름(Exception Path)**을 함께 기술한다.
> 백엔드 구현 관점에서 중요한 처리 포인트는 `🔧` 로 표기한다.

---

### 시나리오 1. 회원가입 및 로그인

**페르소나:** 김지수 (일반 회원)
**맥락:** 처음 TicketFlow에 방문한 김지수가 회원가입 후 로그인한다.

#### Happy Path

```
1. 김지수가 이메일, 비밀번호, 닉네임을 입력하고 회원가입 버튼을 누른다.
2. 🔧 서버는 이메일 중복 여부를 확인한다.
3. 비밀번호를 BCrypt로 암호화하여 DB에 저장한다.
4. 김지수가 이메일·비밀번호를 입력하고 로그인한다.
5. 🔧 서버는 JWT Access Token + Refresh Token을 발급하여 응답한다.
6. 이후 모든 요청에 Authorization 헤더로 Access Token을 포함한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 이미 가입된 이메일 | 409 Conflict 반환 |
| 비밀번호 불일치 | 401 Unauthorized 반환 |
| Access Token 만료 | Refresh Token으로 재발급 |

---

### 시나리오 2. 이벤트 검색 및 인기 검색어 조회

**페르소나:** 김지수 (일반 회원)
**맥락:** 오픈런에 실패한 김지수가 다른 공연을 찾아보기 위해 검색을 사용한다.

#### Happy Path

```
1. 김지수가 검색창에 "아이유"를 입력하고 검색한다.
2. 🔧 서버(v1 API): MySQL LIKE 쿼리로 이벤트를 조회하고 페이징하여 반환한다.
   🔧 서버(v2 API): Caffeine 캐시를 먼저 확인한다.
                   캐시 Hit → 즉시 반환 (DB 조회 없음)
                   캐시 Miss → MySQL 조회 후 캐시 저장 → 반환
3. 🔧 검색어 "아이유"를 Redis ZSet에 ZINCRBY로 점수 +1 처리한다. (실시간 누적)
4. 김지수가 검색 결과 목록을 확인하고 원하는 이벤트를 클릭한다.
5. 인기 검색어 Top 10 API를 호출하면 Redis ZREVRANGE로 실시간 랭킹을 반환한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 검색 결과 없음 | 빈 배열 + 200 OK 반환 |
| 검색어 미입력 | 400 Bad Request 반환 |
| Redis 장애 시 인기 검색어 | fallback으로 빈 리스트 반환, 서비스 중단 없음 |

> **캐싱 포인트 정리**
>
> | API 버전 | 캐시 | 비고 |
> |----------|------|------|
> | v1 `/api/v1/events/search` | 없음 | 매 요청 MySQL 직접 조회 |
> | v2 `/api/v2/events/search` | Caffeine (TTL 5분) | 필수 구현 |
> | v2 업그레이드 | Redis Cache-Aside | 도전 구현 |

---

### 시나리오 3. 선착순 쿠폰 발급

**페르소나:** 김지수 (일반 회원)
**맥락:** 콘서트 오픈 전날, 이민준이 세팅한 10% 할인 쿠폰 100장을 김지수가 발급받으려 한다.

#### Happy Path

```
1. 김지수가 쿠폰 발급 버튼을 누른다.
2. 🔧 서버는 Lettuce SETNX로 쿠폰 분산락을 획득한다.
   Key: lock:coupon:{couponId}, TTL: 3초
3. 🔧 현재 발급 수량을 조회한다. (발급 수 < 100)
4. 🔧 쿠폰 발급 처리 후 발급 수량 +1, DB에 UserCoupon INSERT
5. 🔧 분산락 해제 (UUID 검증 + Lua Script 원자적 삭제)
6. 김지수에게 쿠폰 발급 완료 응답을 반환한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 수량 소진 (100장 완료) | 409 Conflict "쿠폰이 모두 소진되었습니다" |
| 동일 쿠폰 중복 발급 시도 | 409 Conflict "이미 발급받은 쿠폰입니다" |
| 락 획득 실패 (경합) | Retry with backoff 최대 3회 → 실패 시 503 반환 |
| 비로그인 사용자 | 401 Unauthorized |

> **🔧 동시성 시나리오**
> 100장 쿠폰에 500명이 동시에 요청 → Lettuce 분산락으로 순차 처리
> → 정확히 100건만 발급 완료, 나머지 400건은 수량 소진 에러 반환

---

### 시나리오 4. 좌석 선택 및 임시 점유 (Hold)

**페르소나:** 김지수 (일반 회원)
**맥락:** 콘서트 오픈 직후 김지수가 A구역 1열 1번 좌석을 선택한다.

#### Happy Path

```
1. 김지수가 좌석 배치 화면에서 A구역 1열 1번(AVAILABLE 상태)을 클릭한다.
2. 🔧 서버는 Lettuce SETNX로 좌석 분산락을 획득한다.
   Key: lock:seat:{eventId}:{seatId}, TTL: 10초
3. 🔧 해당 좌석의 Hold 키 존재 여부를 확인한다. (미존재 → 점유 가능)
4. 🔧 Redis에 Hold 키를 SET한다.
   Key: hold:{eventId}:{seatId}, Value: userId, TTL: 5분
5. 🔧 user-holds:{userId} Set에 seatId를 추가한다. (어뷰징 방지)
6. 🔧 분산락 해제
7. 김지수에게 holdToken을 발급하고 결제 화면으로 이동시킨다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 이미 다른 사용자가 Hold 중인 좌석 | 409 Conflict "이미 선점된 좌석입니다" |
| 동일 사용자 4석 이상 Hold 시도 | 400 Bad Request "최대 4석까지 선택 가능합니다" |
| Hold TTL 5분 만료 | Redis 자동 삭제 → 좌석 AVAILABLE 복원 |
| 비로그인 사용자 | 401 Unauthorized |

---

### 시나리오 5. 결제 및 예매 확정

**페르소나:** 김지수 (일반 회원)
**맥락:** 좌석 Hold 후 쿠폰을 적용하고 결제를 완료한다.

#### Happy Path

```
1. 김지수가 결제 화면에서 보유 쿠폰(10% 할인)을 선택한다.
2. 김지수가 결제 버튼을 누른다. (holdToken 포함)
3. 🔧 서버는 Mock PG에 결제 요청을 보낸다.
4. 🔧 Mock PG가 결제 완료 웹훅을 /api/mock-pg/webhook으로 전송한다.
5. 🔧 서버는 웹훅 수신 후 Hold 유효성을 검증한다.
   - Redis에 hold:{eventId}:{seatId} 키가 존재하는가? (TTL 만료 여부 확인)
6. 🔧 [TTL 유효] 락 획득 순서 준수하여 처리:
   ① 좌석 Hold 락 재획득 (Lettuce)
   ② 쿠폰 사용 락 획득 (Lettuce)
   ③ DB 트랜잭션 시작
      - UserCoupon.status = USED
      - Booking INSERT (status = CONFIRMED)
      - Redis Hold 키 삭제
   ④ DB 트랜잭션 커밋
   ⑤ 락 역순 해제 (쿠폰 → 좌석)
7. 김지수에게 예매 확정 응답 및 예매번호를 반환한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 웹훅 수신 시 Hold TTL 만료 | 예매 실패 처리, 결제 취소 요청 (Mock PG) |
| 쿠폰 이미 사용됨 | 쿠폰 없이 재시도 안내 후 예매 실패 |
| 결제 금액 불일치 | 400 Bad Request, 예매 취소 |
| 웹훅 중복 수신 (idempotency) | 예매번호 중복 체크 후 무시 |

---

### 시나리오 6. 내 예매 내역 조회

**페르소나:** 김지수 (일반 회원)
**맥락:** 예매 후 예매 내역을 확인하고 싶다.

#### Happy Path

```
1. 김지수가 마이페이지 → 예매 내역 탭에 접근한다.
2. 🔧 서버는 JWT에서 userId를 추출하여 본인 예매 목록을 조회한다.
3. 예매 상태(CONFIRMED / CANCELLED), 이벤트명, 좌석 정보, 결제 금액을 반환한다.
4. 김지수가 특정 예매 상세를 클릭하면 상세 정보를 조회한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 예매 내역 없음 | 빈 배열 + 200 OK |
| 타인의 예매 상세 접근 | 403 Forbidden |

---

### 시나리오 7. CS 문의 채팅

**페르소나:** 김지수 (일반 회원) ↔ 이민준 (관리자)
**맥락:** 예매 후 좌석 정보가 잘못 표시된 것 같아 김지수가 CS 문의를 개설한다.

#### Happy Path (김지수 — 문의 개설)

```
1. 김지수가 CS 문의하기 버튼을 클릭하고 문의 제목을 입력한다.
2. 🔧 서버는 ChatRoom을 생성한다. (status: WAITING, 개설자: 김지수)
3. 김지수가 WebSocket(STOMP)으로 채팅방에 접속한다.
   🔧 STOMP CONNECT 시점에 ChannelInterceptor가 JWT를 검증하고 Principal 설정
4. 김지수가 메시지를 전송한다.
   🔧 /pub/chat/{roomId}로 SEND → 서버가 DB에 메시지 저장 → /sub/chat/{roomId}로 브로드캐스트
5. 김지수에게 "문의가 접수되었습니다" 시스템 메시지가 전달된다.
```

#### Happy Path (이민준 — 문의 처리)

```
1. 이민준이 관리자 페이지에서 CS 문의 목록을 조회한다.
   🔧 전체 ChatRoom 목록 조회 (status별 필터: WAITING / IN_PROGRESS / COMPLETED)
2. 이민준이 WAITING 상태인 김지수의 문의 채팅방에 입장한다.
3. 🔧 ChatRoom.status를 WAITING → IN_PROGRESS로 변경한다.
4. 이민준이 답변 메시지를 전송한다. → 김지수에게 실시간 전달
5. 처리 완료 후 이민준이 문의 상태를 IN_PROGRESS → COMPLETED로 변경한다.
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 일반 회원이 문의 상태 변경 시도 | 403 Forbidden |
| 일반 회원이 다른 회원의 채팅방 접근 | 403 Forbidden |
| WebSocket 연결 끊김 | 클라이언트 재연결, lastMessageId 기반 미수신 메시지 복구 |
| JWT 만료 상태로 STOMP CONNECT | 연결 거부, 401 반환 |

> **채팅 상태 전이 규칙**
>
> ```
> WAITING → IN_PROGRESS → COMPLETED
> ```
> - 역방향 전이 불가 (COMPLETED → WAITING 불허)
> - 상태 변경 권한: 관리자만 가능

---

### 시나리오 8. 이벤트 등록 (관리자)

**페르소나:** 이민준 (관리자)
**맥락:** 새로운 콘서트 정보를 시스템에 등록한다.

#### Happy Path

```
1. 이민준이 이벤트 등록 폼에 공연명, 장르, 날짜, 장소, 구역·좌석 정보를 입력한다.
2. 🔧 서버는 ADMIN 권한을 확인한다. (JWT role 검증)
3. 🔧 Event, Venue, Section, Seat 엔티티를 순서대로 저장한다.
4. 이민준에게 등록 완료 응답과 이벤트 ID를 반환한다.
5. 이민준이 해당 이벤트에 선착순 쿠폰을 추가로 생성한다.
   🔧 Coupon(totalQuantity, issuedQuantity=0) INSERT
```

#### Exception Path

| 상황 | 서버 처리 |
|------|-----------|
| 일반 회원이 이벤트 등록 시도 | 403 Forbidden |
| 필수 항목 누락 | 400 Bad Request + 누락 필드 안내 |
| 좌석 수량이 0 이하 | 400 Bad Request |

## 3. 시나리오 흐름 요약

```
[김지수 — 일반 회원 전체 흐름]

회원가입/로그인
    │
    ▼
이벤트 검색 (v1/v2) + 인기 검색어 조회
    │
    ▼
쿠폰 발급 (선착순, 동시성 제어)
    │
    ▼
좌석 선택 → Hold 획득 (5분 TTL)
    │
    ├── Hold 실패 → 다른 좌석 선택
    │
    ▼
쿠폰 적용 + Mock PG 결제
    │
    ├── TTL 만료 → 예매 실패 → 처음부터
    │
    ▼
예매 확정 (웹훅 수신 → DB 저장)
    │
    ▼
내 예매 내역 조회
    │
    ▼
CS 문의 채팅 (필요 시)


[이민준 — 관리자 전체 흐름]

이벤트 등록 + 좌석/구역 설정
    │
    ▼
쿠폰 생성 (수량·할인율 설정)
    │
    ▼
CS 문의 목록 모니터링
    │
    ▼
문의 채팅방 입장 → 답변 → 상태 완료 처리
```

## 4. 시나리오별 핵심 기술 매핑

| 시나리오 | 핵심 기술 | 구현 단계 |
|----------|-----------|-----------|
| 로그인 | JWT (Access + Refresh Token) | 필수 |

| CS 채팅 | WebSocket + STOMP + ChannelInterceptor | 도전 |
