## 반영 결정 요약

| 항목 | 분류 | 결정 | 상태 |
|------|------|------|------|
| C-01. SEAT.status vs ACTIVE_BOOKING 참조 불일치 | 교차 문서 불일치 | 허용 — API 명세서를 ACTIVE_BOOKING 기반으로 통일 | ✅ 반영 |
| C-02. TICKET 테이블 누락 | 교차 문서 불일치 | B안 — BOOKING이 티켓 역할 겸임, `ticket_code` 컬럼 추가 | ✅ 반영 |
| C-03. Hold 해제 응답 코드 (200 vs 204) | 교차 문서 불일치 | 200 OK + JSON body `{ message, seatId }` 로 통일 | ✅ 반영 |
| C-04. Lock 실패 에러 코드 불일치 | 교차 문서 불일치 | 409 `SEAT_LOCK_FAILED` 로 통일 | ✅ 반영 |

| C-06. 검색 파라미터명 불일치 (keyword vs q) | 교차 문서 불일치 | `keyword` 로 통일 | ✅ 반영 |

### 4. 기능명세서 (v3.0 → v4.0)

#### 변경 내용

**① FN-BK-01: 주문 생성 시 Hold TTL 연장 (L-02 대응)**

**문제 상황:**
```
Hold TTL = 5분
주문 생성 시점에 Hold 남은 시간 = 30초
→ PG 웹훅 도착 시점에 Hold TTL 만료
→ 사용자는 결제했지만 예매 실패 (최악의 UX)
```

**해결책:**
```
주문 생성 성공 직후, 모든 hold:{eventId}:{seatId} 키에 EXPIRE 300 재실행
→ TTL이 다시 5분으로 초기화
→ 주문 생성~웹훅 수신 구간에서 TTL 만료 문제 해소
```

처리 로직에 6번 단계로 추가:
```
6. Hold TTL 연장: 모든 hold:{eventId}:{seatId} 키에 EXPIRE 300 재실행 (+5분)
```

**② FN-BK-02: TICKET 발급 → ticketCode 생성으로 변경 (C-02 B안)**

별도 TICKET 테이블을 만드는 대신 BOOKING.ticket_code 컬럼에 저장한다.

```
[변경 전]
f. 각 BOOKING별 TICKET 발급

[변경 후]
f. 각 BOOKING.ticket_code 생성 및 저장
   (형식: "TF-" + 연도 + 구역약자 + 좌석번호 + UUID 앞 4자리)
```

**③ FN-BK-03: 취소 플로우에 SELECT ... FOR UPDATE 추가 (L-04 대응)**

**문제 상황:** 취소 트랜잭션이 진행되는 동안 다른 사용자의 웹훅 확정이 동일 좌석에 도달하면 데이터 정합성이 깨진다.

**해결책:** 취소 트랜잭션 진입 시 관련 행을 FOR UPDATE로 선점.

```sql
-- 취소 트랜잭션 내 비관적 락
SELECT * FROM booking WHERE order_id = ? FOR UPDATE;
SELECT * FROM `order` WHERE order_id = ? FOR UPDATE;
SELECT * FROM active_booking WHERE seat_id IN (...) FOR UPDATE;
```

처리 순서도 재정렬: `ACTIVE_BOOKING DELETE → BOOKING CANCELLED → ORDER CANCELLED` (정합성 순서 보장)

**④ FN-CPN-02: 쿠폰 이중 잠금 제거 (L-01 대응)**

**문제 상황:** Redis DECR 자체가 원자적인데 분산락까지 이중으로 걸고 있었음.

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 수량 차감 | 분산락 + Redis DECR (이중) | Redis DECR + Lua Script만 (단일) |
| 락 실패 처리 | 503 반환 | DECR 실패 → 즉시 409 |
| DB fallback | 미명시 | Redis 장애 시 `SELECT ... FOR UPDATE on coupon` |

**Lua Script 방어 로직:**
```lua
-- 음수 방어: DECR 후 0 미만이면 원상복구
local val = redis.call('DECR', KEYS[1])
if val < 0 then
  redis.call('INCR', KEYS[1])
  return -1  -- 소진 신호
end
return val
```

| 선택지 | 내용 | 결정 |

### 6. API 명세서 (v3.0 → v4.0)

#### 변경 내용

**① C-01: 좌석 조회 설명 — ACTIVE_BOOKING 기반으로 통일**

```
[변경 전]
좌석 상태 = DB `seat.status`(CONFIRMED) + Redis `hold:{eventId}:{seatId}` 존재 여부(ON_HOLD) 합산

[변경 후]
좌석 상태 판단 로직 (ERD v7.0 기준):
1. ACTIVE_BOOKING 테이블에 해당 seat_id 존재 → CONFIRMED
2. Redis hold:{eventId}:{seatId} 키 존재 → ON_HOLD
3. 그 외 → AVAILABLE
```

**② C-02: 웹훅 응답 tickets[] → bookings[]로 변경**

```json
// 변경 전
"tickets": [
  { "ticketId": 201, "ticketCode": "TF-2026-A015-XYZ1", "seatNumber": "A구역-015" }
]

// 변경 후
"bookings": [
  { "bookingId": 201, "ticketCode": "TF-2026-A015-XYZ1", "seatInfo": "A구역 A열 15번" }
]
```

**③ C-05: venueId 예시 데이터 수정**

모든 API 응답 예시의 `"venueId": 5` → `"venueId": 1` 로 수정 (단일 공연장 고정 원칙 준수).

**④ 좌석 조회 API에 sectionId 쿼리 파라미터 추가**

```
[변경 전]
GET /api/events/{eventId}/seats | Request: —

[변경 후]
GET /api/events/{eventId}/seats | Request: ?sectionId (선택)
```

**⑤ 인기 검색어 클릭 API 신규 추가 (ZINCRBY)**

화면 설계서에는 있었으나 API 명세서에 누락되어 있던 클릭 이벤트 처리 API를 추가.

```
POST /api/search/popular/click
Request: { "keyword": "세븐틴" }
동작: Redis ZINCRBY search-keywords 1 {keyword}
```

**⑥ L-01: 쿠폰 발급 동시성 제어 설명 수정**

```
[변경 전]
Redis Atomic DECR + Lua Script (Primary 수량 관리)
+ Redis 분산락 lock:coupon:{couponId} [이중 잠금]

[변경 후]
Redis Atomic DECR + Lua Script (DECR 자체가 원자적 — 추가 분산락 불필요)
Redis 장애 시 DB SELECT ... FOR UPDATE로 fallback
```

**⑦ L-02: 주문 생성 API에 Hold TTL 연장 명시**

```
POST /api/orders 설명에 추가:
Hold TTL 연장 (L-02 대응): 주문 생성 성공 시 모든 Hold 키의 TTL을 +5분 연장
Redis EXPIRE hold:{eventId}:{seatId} 300 재실행
```

**⑧ L-04: 주문 취소 API에 SELECT ... FOR UPDATE 명시**

```
POST /api/orders/{orderId}/cancel 설명에 추가:
동시성 제어: SELECT ... FOR UPDATE로 BOOKING, ORDER, ACTIVE_BOOKING 행 잠금
단일 트랜잭션 내 원자적 처리:
  1. ACTIVE_BOOKING DELETE
  2. BOOKING → CANCELLED
  3. ORDER → CANCELLED
  4. USER_COUPON → ISSUED (쿠폰 복원)
```

**⑨ 관리자 채팅방 목록 API 추가 (L-03)**

```
GET /api/admin/chat/rooms?status=OPEN&page=0&size=20
인증: ADMIN 권한
용도: 관리자 대시보드 초기 로딩 시 OPEN 채팅방 목록 조회 (WebSocket은 신규 알림 전용)
```
