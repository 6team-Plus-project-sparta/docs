# 📋 SA 문서 변경 사항 (v4 → v5)

> **작성일:** 2026-04-09
> **변경 범위:** 2. 사용자 시나리오 / 3. 유스케이스 명세서 / 4. 기능 명세서 / 5. ERD

## 5. ERD (v4 → v5)

| 분류 | 항목 | 변경 내용 |
|------|------|-----------|
| 테이블 삭제 | `ORDER_ITEM` | 삭제. 역할을 `BOOKING`이 흡수 |
| 테이블 신규 | `ACTIVE_BOOKING` | 확정 예약만 보관. `seat_id PK` + `booking_id UK`. 취소 시 DELETE, 이력은 BOOKING에 남김 |
| BOOKING 컬럼 추가 | `order_id FK` | 소속 주문 연결 (ORDER_ITEM 역할 흡수) |
| BOOKING 컬럼 추가 | `original_price` | 결제 당시 원가 스냅샷 (ORDER_ITEM 역할 흡수) |
| BOOKING 컬럼 추가 | `event_id FK` | 물리적 공간 내 어떤 이벤트인지 구별 |
| BOOKING 컬럼 추가 | `status` | PENDING / CONFIRMED / CANCELLED / FAILED |
| SEAT 컬럼 삭제 | `status` | 삭제. CONFIRMED 판단은 ACTIVE_BOOKING 존재 여부로 대체 |
| SEAT 컬럼 삭제 | `seat_number` | 삭제. `row_name` + `col_num` 조합으로 대체 |
| EVENT 컬럼 추가 | `round_number` | 다회차 회차 번호. "공연 1건 = 회차 1건" 원칙 유지 |
| EVENT 컬럼 추가 | `status` | ON_SALE / SOLD_OUT / CANCELLED / ENDED |
| VENUE 컬럼 삭제 | `capacity` | 삭제. 잔여 좌석은 SEAT 기반 동적 계산으로 대체 |
| 관계 변경 | `ORDER → BOOKING` | `ORDER ||--o{ BOOKING` (ORDER_ITEM 제거로 직접 연결) |
| 관계 신규 | `ACTIVE_BOOKING` | `ACTIVE_BOOKING ||--|| SEAT`, `ACTIVE_BOOKING ||--|| BOOKING` |
| 도메인 경계 | Booking/Order Context | `ORDER_ITEM` 제거, `ACTIVE_BOOKING` 추가 |
| 인덱스 표 | `BOOKING` | `(seat_id, status)`, `order_id` 인덱스 추가. `ORDER_ITEM` 관련 항목 제거 |

## 3. 유스케이스 명세서 (v1 → v2)

| UC | 항목 | 변경 내용 |

| UC-007 | 주요 흐름 4번 | 좌석 상태 확인을 `Redis Hold 키 + ACTIVE_BOOKING 존재 여부`로 변경 |
| UC-007 | 비고 추가 | Hold 수 Race Condition 설명 및 Lua Script 원자적 구현 명시 |
| UC-008 | 주요 흐름 전면 재작성 | 피드백 기준 13단계 플로우로 재정렬 |
| UC-008 | 주요 흐름 5번 | `ORDER_ITEM + BOOKING` → `BOOKING` 생성으로 단순화 |
| UC-008 | 주요 흐름 10번 | DB 트랜잭션 내 `ACTIVE_BOOKING INSERT` 명시 |
| UC-008 | 예외 흐름 `10a` 추가 | ACTIVE_BOOKING INSERT 시 seat_id 중복 키 에러 → 롤백 (DB 최종 방어선) |
| UC-009 | 주요 흐름 3번 | `SEAT.status → AVAILABLE` → `ACTIVE_BOOKING DELETE`로 변경 |
| UC-010 | 주요 흐름 3번 | `ORDER_ITEM` → `BOOKING` 목록 기준 조회로 변경 |
| UC-011 | 주요 흐름 `1-1` 추가 | 락 획득 전 중복 발급 체크 단계 명시 |
| UC-011 | 대체 흐름 위치 정정 | 중복 발급 → `1-1a` (락 획득 전), 수량 소진 → `3a` (락 내부) |

## 4. 기능 명세서 (v2 → v3)

| FN | 항목 | 변경 내용 |
|----|------|-----------|
| FN-EVT-01 | 입력값 | `roundNumber`, `status`, `rowCount`, `colCount` 추가 |
| FN-EVT-01 | 처리 로직 3번 | Event 저장 시 `roundNumber`, `status` 포함 명시 |
| FN-EVT-01 | 처리 로직 5번 | Seat 생성 기준을 `rowCount × colCount` 로 변경 |
| FN-EVT-02 | 입력값 | `status` 필터 파라미터 추가 (기본값: ON_SALE) |
| FN-EVT-02 | 처리 로직 | `status` 필터 조건 추가 |
| FN-EVT-03 | 처리 로직 3번 | 잔여 좌석 계산을 `ACTIVE_BOOKING NOT EXISTS` 서브쿼리로 변경 |
| FN-SEAT-01 | 처리 로직 | 좌석 상태 판단 우선순위 변경: ACTIVE_BOOKING → Redis → AVAILABLE |
| FN-SEAT-01 | 응답 JSON | `seatNumber` → `rowName` + `colNum` 으로 변경 |
| FN-SEAT-01 | 비고 추가 | CONFIRMED 판단 기준 명시 (ACTIVE_BOOKING 존재 여부) |
| FN-SEAT-02 | 처리 로직 4번 | `seat.status` 확인 → `ACTIVE_BOOKING` 존재 여부 확인으로 변경 |
| FN-BK-01 | 처리 로직 5번 | `ORDER_ITEM + BOOKING` → `BOOKING` 생성. `order_id`, `original_price`, `event_id` 포함 명시 |
| FN-BK-02 | 처리 로직 4번 | 트랜잭션 내 `ACTIVE_BOOKING INSERT` 추가. 중복 키 에러 시 롤백(DB 최종 방어선) 명시 |
| FN-BK-03 | 처리 로직 4번 | 취소 트랜잭션에 `ACTIVE_BOOKING DELETE` 추가 |
| FN-BK-04 | 처리 로직 | `ORDER_ITEM` → `BOOKING` 기준 조회로 변경 |
| FN-BK-04 | 응답 JSON | `items` → `bookings`, `seatNumber` → `rowName` + `colNum` |
| FN-CPN-02 | 처리 로직 2번 | 중복 발급 체크 위치를 락 획득 **전**으로 이동 |
| FN-CPN-02 | 비고 추가 | 중복 체크 선행 근거 설명 (이미 받은 사용자 빠른 차단, 부하 감소) |
| FN-SRCH-05 | 최적화 대상 쿼리 | `ORDER_ITEM` → `BOOKING` 기준으로 수정 |
| FN-SRCH-05 | 추천 인덱스 | `ORDER_ITEM` 인덱스 제거, `BOOKING` 인덱스(`order_id`, `seat_id + status`) 추가 |
| 핵심 설계 원칙 | 전면 수정 | `ORDER_ITEM` 제거, `ACTIVE_BOOKING` 역할, `SEAT.status` 삭제 내용 반영 |

## 2. 사용자 시나리오 (v1 → v2)

| 페르소나 | 항목 | 변경 내용 |
|----------|------|-----------|
| 박서준 | 시나리오 1-1 | `BOOKING CONFIRMED + ACTIVE_BOOKING INSERT` 반영 |
| 박서준 | **시나리오 1-4 신규 추가** | 멀티 좌석(4석) 선택 → 일괄 결제 플로우. Hold Set 관리, ORDER ↔ BOOKING 1:N 구조, 어뷰징 방지 검증 포인트(5석 시도 시 차단) 포함 |
| 이도현 | 시나리오 3-1 | 중복 발급 체크 선행 플로우 반영 (락 바깥에서 먼저 체크) |
| 이도현 | 시나리오 3-2 | `ACTIVE_BOOKING INSERT` 명시 |
| 매핑 매트릭스 | 항목 추가 | `멀티 좌석 Hold (최대 4석)`, `ACTIVE_BOOKING (중복 확정 방어)` 추가 |
